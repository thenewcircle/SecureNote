Storage

  • Improper storage of data is a simple problem to resolve, if handled from the start
  • Mobile devices are different than traditional computing systems
    • Physical device access is easy to achieve
    • Once physical access is available, device can be rooted, flashed, etc.
  • You must assume that once data is stored on the device it is vulnerable!
    • For typical users, this is usually not a problem
    • Malicious users are not typical users!
      • Once a device has been rooted, re-ROM’d, etc. the malicious user can read/write anywhere they want

        The question to ask: what does my app really need to store?

  • The general rule of thumb: store only what you absolutely require
    • This can mean many things: tokens/keys to access APIs, user credentials, authentication tokens
    • It boils down to propery managing sensitive data
  • If you must store sensitive data, have a plan to do so in a way with minimized risk

Storage APIs

  • File access
    • Context provides a number of helper APIs
      • Context.openFileInput() opens a private file for reading
      • Context.openFileOutput() opens a private file for writing
      • Private files are tied to the app package, located in its private sandbox
      • There are additional APIs for cache files
      • "External storage" is also available, with a package private area
    • Standard Java File class and related APIs available
      • Context provides APIs to help get package private paths (getDir(), getCacheDir(), getExternalCacheDir(), getExternalFilesDir(), getFilesDir())
      • Apps should not assume any specific filesystem paths or mount points
  • Opaque Binary Blobs (OBB)
    • App specific file container
    • Can be encrypted
    • App manages the download / mount of the containers
    • Useful for large data sets (graphics, assets)
    • Use StorageManager to mount then access
    • Cannot create new OBBs on the device, just access existing ones
  • SharedPreferences
    • File backed storage of key-value pair data
      • Keys are String objects, values can be primitives, String or Set<String>
    • App controls what data is stored and meaning
    • Default for the app is tied to the package name
      • Default name can be overridden
      • Can use multiple SharedPreferences for the same app
    • Data is not encrypted, simple string based lookup
    • Can be used by multiple threads (clients), modifications go through SharedPreferences.Editor
    • Does not support multi-process access
  • SQLite Databases
    • SQLite databases can be used for app specific storage
      • Accessible to any component in your app, but (generally) not outside your app
      • Backed by private file within the sandbox
    • Recommended use is via SQLiteOpenHelper class to help manage creation, updates
      • Use SQLiteOpenHelper.getWriteableDatabase() and SQLiteOpenHelper.getReadableDatabase() to get a SQLiteDatabase object to call query(), etc. against
      • query() calls return Cursor objects which points to the rows found by the query

User Authentication

  • User authentication is used for access to user specific data in the app
  • Best Practices for storage depends on app and credentials
    • BP#1: Certificate/key based credentials
      • Leverage Android’s credential storage to securely track
        • See KeyChain and AndroidKeyStore provider with KeyStore (Android 4.3 and newer)
      • DO NOT assume filesystem sandbox is enough
      • Still can be vulnerable to attacks if device compromised
    • BP#2: User account credentials
      • Users have numerous accounts for different service (Google, PayPal, Facebook, etc.)
      • Leverage AccountManager for storage of accounts and accessing accounts
        • AccountManager uses plugabble architecture for authenticators
        • Using an existing account, apps can request authentication token for use in requests
      • DO NOT assume filesystem sandbox is enough and store in cleartext
    • BP#3: Use WebView and Session Tokens
      • Present the user with web page for login
      • DO NOT store the user credentials at all
      • The web interface presents a redirect with session token on success (e.g. OAuth 2.0)
      • Keep the session token for subsequent requests
      • Session key can be stored, but carefully do so
      • This relies on the server to properly handle credentials, expire and rotate tokens correctly

Session Identifiers

  • Many network services (Google, Facebook, Instagram, etc.) use session or token based identifiers for access
  • These can be stored on the device, carefully
    • Be sure to store them in a way so an attacker cannot determine how it is used
  • Servers should have reasonable expiration times for session IDs
    • Some server implementations may issue new session IDs to replace old ones frequently
  • Android’s AccountManager can be leveraged with 3rd party authenticators to help with auth tokens or sessions
    • It is up to the app to notify the AccountManager if a token is no longer valid

Hardcoded Tokens and Keys

  • Best practice is to not use hardcoded values in code
    • Java (including Dalvik DEX) can be disassembled very easily
    • Reverse engineering can discover hardcoded values relatively easily
    • This is particularly bad for encryption of data
    • Best Practice: hash a random salt plus and password to create a key (with numerous iterations), store the salt and hash
  • Some cloud services require an app authentication token
    • You have no choice but to store or hard code
    • Best practice: ensure the server only uses this to "authenticate" the app, not the user
    • Best practice on the server: only allow a limited subset of functionality via the app (e.g. no account delete or other more volatile ops)

Unique Identifiers

  • Use of unique IDs is common for apps to track users or installs
    • Typically used for analytics, ads, etc.
  • A common practice uses TelephonyManager.getDeviceId()
    • DO NOT do this! There are better ways.
    • It requires the permission READ_PHONE_STATE which can set up a red flag to users
    • It is not reliable. Some devices do not report unique IDs or change IDs after factory reset
  • Best Practice: for non-physical device tracking use UUID.randomUUID()
    • If you must track physical devices, use Settings.Secure.ANDROID_ID
      • This is a 64 bit value created and stored when the device first boots. It is wiped at factory reset.
      • The downside to this is it is unreliable before Android 2.3 (Gingerbread) due to SecureRandom used to create ANDROID_ID being seeded incorrectly.
  • See this Google Developer Blog for more details

File Usage

  • Regular file usage is simple, well understood and fairly robust
  • SharedPreferences are backed by files, so the same concepts apply (except for encryption)
  • Inherently insecure at runtime or if device physically compromised
    • Data is "raw" or in the clear
    • App sandbox works for basic protection
      • Easy to override
    • External storage can present issues, if removable
  • For simple data (not critical if exposed), basic File access is ok
  • Best Practice for sensitive data: use encryption (file only, not SharedPreferences) with user provided password
    • No "encrypt file" direct API available, but javax.crypto package can be used
      • Use a Cipher initilized for AES/CBC/PKCS5Padding, create an initialization vector (IV) using SecureRandom, save the raw IV at the start of the file, then use CipherOutputStream to write the encrypted data to the file. Use the password to derive a key (don’t store the password!) using PBKDF2.
      • To ensure integrity of the crypted file, create a secure checksum with MAC or HMAC (i.e. SHA-1)
      • Obviously, this is not simple, can be prone to error

SQLite Databases

  • SQLite databases are used frequently in Android
  • Each database is backed by a file
  • Inherently insecure at runtime or if device physically compromised
    • App sandbox works for basic protection
    • Easy to override
  • For simple data (i.e. no user details, password hashes, etc.) basic usage is ok
  • Best Practice for permissions: don’t override sandbox settings
  • Best Practice for location: don’t use external (removable) storage
  • Best Practice for sensitive data: use encryption (SQLCipher)

SQLCipher

  • Crypto library for SQLite databases, made by Zetetic LLC
  • Encrypts 100% of the database, with a small amount of performance overhead (claims 5-15% overhead)
  • Utilizes industry standard crypto libraries from OpenSSL, utlizes 256-bit AES encryption
    • Encryption keys can be derived from a passphrase or using raw binary key data
    • Key derivation is done using a 16 byte secure random salt (OpenSSL’s RAND_bytes)
    • Each "page" in the database is encrypted separately
  • Multi-platform and language support: Android, Linux, Mac OSX, iOS, and Windows with C/C++, Java, .NET, Ruby, PHP, Python
  • Has both an open source (Community Edition) and commercial licensed versions
    • Community Edition is available via BSD-style license
    • Android Community Edition binaries are available free of charge at https://s3.amazonaws.com/sqlcipher/SQLCipher+for+Android+v3.1.0.zip
  • Basic Android walk-thru instructions (assumes Eclipse/ADT, not Android Studio) available here
    • In code, the major change is the import and use of net.sqlite.database.SQLiteDatabase rather than android.database.sqlite.SQLiteDatabase

Remembering Users with Accounts

  • Given the potential dangers around applications leaking persisted data on the file system, one option for storing user credentials is the Accounts Framework.
  • Accounts allow the system to manage access to credentials and authorization tokens for web services.
  • This presents two useful options to application developers:

    1. Use an existing account on the system (like the user’s Google account) rather than inventing your own.

    2. Add your own account authentication logic (via a custom AccountAuthenticator) to the framework to get the credentials storage out of the application space.

  • The following is an example of how an application can use the AccountManager service to authenticate a user using their Google account:
    The following method will work for accounts from any active service (e.g. Twitter, Facebook, Microsoft Exchange) that use the accounts framework. For Google accounts specifically, Google Play Services now provides a much simpler API.

    1. Accessing the accounts framework requires two permissions: 

      <uses-permission android:name="android.permission.GET_ACCOUNTS" />
<uses-permission android:name="android.permission.USE_CREDENTIALS" />

    2. You can ask the user to choose a Google account for authentication: 

      static final int REQUEST_CODE_PICK_ACCOUNT = 1000;

private void pickUserAccount() {
    String[] accountTypes = new String[]{"com.google"};
    Intent intent = AccountManager.newChooseAccountIntent(
            null,          // Currently selected account
            null,          // List of allowed accounts
            accountTypes,  // Allowed account types
            true,          // Show the picker UI
            null,
            null,
            null,
            null);
    startActivityForResult(intent, REQUEST_CODE_PICK_ACCOUNT);
}

    3. Determine the name of the account they chose, then find the Account object that matches the selected name from the AccountManager: 

      @Override
protected void onActivityResult(int requestCode, int resultCode,
        Intent data) {
    if (requestCode == REQUEST_CODE_PICK_ACCOUNT) {
        // Receiving a result from the AccountPicker
        if (resultCode == RESULT_OK) {
            String accountName =
                    data.getStringExtra(AccountManager.KEY_ACCOUNT_NAME);

            //Find the matching account for the selected name
            AccountManager accountManager = AccountManager.get(this);
            Account[] accounts =
                    accountManager.getAccountsByType("com.google");
            for (Account account : accounts) {
                if (account.name.equals(accountName)) {
                    //Found a match, get an authentication token
                    getToken(account);
                    break;
                }
            }
        }
    }
    …
}

    4. Using AccountManager.getAuthToken(), we can ask the accounts framework to asynchronously authorize the user: 

      static final String SCOPE =
        "oauth2:https://www.googleapis.com/auth/userinfo.profile";

public void getToken(Account selectedAccount) {

      • Otherwise, the token will be returned in the callback Bundle.

Encryption

  • Enhanced privacy and integrity of data (beyond the application sandbox) can be achieved using encryption
  • On-device data encryption is usually done via Java Cryptography Extension (JCE)
  • The key to secure on-device encryption is not storing the decryption keys on the file system
    • If the file system is compromised (e.g. rooted) the keys are just as easy to obtain as the data
  • The chosen method of encryption should rely on information that cannot be obtained by dumping /data
  • The workhorse of encryption/decryption is the Cipher:
    • To encrypt to a byte array: 
      byte[] plainText = // some plain text
Key key = // derived key for chosen algorithm
Cipher cipher = Cipher.getInstance(ENCRYPTION_ALGORITHM);
cipher.init(Cipher.ENCRYPT_MODE, key);
byte[] encrypted = cipher.doFinal(plainText);
    • To decrypt the same byte array: 
      byte[] encrypted = // read encrypted buffer from somewhere
Cipher cipher = Cipher.getInstance(ENCRYPTION_ALGORITHM);
cipher.init(Cipher.DECRYPT_MODE, key);
byte[] plainText = cipher.doFinal(encrypted);
  • To avoid keeping large blocks of data in memory, there’s CipherInputStream and CipherOutputStream:
    • To encrypt to a file/stream: 
      byte[] plainText = // some plain text
Key key = // derived key for chosen algorithm
OutputStream out = new FileOutputStream("some-file");
try {
    Cipher cipher = Cipher.getInstance(ENCRYPTION_ALGORITHM);
    cipher.init(Cipher.ENCRYPT_MODE, key);
    out = new CipherOutputStream(out, cipher);
    out.write(plainText);
    out.flush();
} finally {
    out.close();
}
    • To decrypt the same file/stream: 
      InputStream in =  new FileInputStream("some-file");
try {
    Cipher cipher = Cipher.getInstance(ENCRYPTION_ALGORITHM);
    cipher.init(Cipher.DECRYPT_MODE, key);
    in = new CipherInputStream(in, cipher);
    in.read(plainText); // do properly
} finally {
    in.close();
}

Password Based Encryption

  • One secure method for encrypting data is to rely on a user-entered password to derive the encryption key.
  • This type of encryption pairs well with the Advanced Encryption Standard (AES) cipher algorithm.
  • AES is a symmetric-key algorithm, meaning it uses the same key to encrypt/decrypt data.
  • Advantage: Key is not stored anywhere an attacker can search without "shoulder surfing"
  • Disadvantage: Requires asking user for passcode repeatedly, or caching in memory (reducing the security factor).
  • We can use the following steps to derive a password-based key:

    1. Android only supports the "PBKDF2WithHmacSHA1" key derivation algorithm, which requires a random salt to apply to the password (to reduce brute-force capabilities).

    2. The Cipher Block Chaining (CBC) mode is well-supported for AES on Android, which also requires a random initialization vector for encryption.

    3. We can generate all the necessary bits with the help of SecureRandom: 

      Cipher cipher = Cipher.getInstance("AES/CBC/PKCS5Padding");

// Do *not* seed secureRandom! Automatically seeded from system entropy.
SecureRandom random = new SecureRandom();

byte[] salt = new byte[32];
random.nextBytes(salt);

byte[] iv = new byte[cipher.getBlockSize()];
random.nextBytes(iv);

    4. The salt will be necessary to derive the SecretKey: 

      private SecretKey generateSecretKey(char[] passphraseOrPin, byte[] salt)
        throws NoSuchAlgorithmException, InvalidKeySpecException {
    // Number of PBKDF2 hardening rounds to use. Larger values increase
    // computation time. You should select a value that causes computation
    // to take >100ms.
    final int iterations = 1000;

    SecretKeyFactory secretKeyFactory =
            SecretKeyFactory.getInstance("PBKDF2WithHmacSHA1");
    KeySpec keySpec =
            new PBEKeySpec(passphraseOrPin, salt, iterations, KEY_LENGTH);
    byte[] keyBytes = secretKeyFactory.generateSecret(keySpec).getEncoded();

    return new SecretKeySpec(keyBytes, "AES");
}
      See Using Cryptography to Store Credentials Safely

    5. This key is used to initialize the Cipher and encrypt some plain text into ciphertext: 

      String passphrase = // user entered password
byte[] plainText = // some plain text
SecretKey key = generateSecretKey(passphrase.toCharArray(), salt);
cipher.init(Cipher.ENCRYPT_MODE, key, new IvParameterSpec(iv));

//Pack the result in a cipher text blob
final byte[] encrypted = cipher.doFinal(plainText);
  • We can use the same process to decrypt the ciphertext data:
    The same salt and initialization vector are required for decrypting the data, so they must be saved somewhere. These values are NOT secret, so it is safe to persist them. 
    final byte[] salt = // retrieve from encryption step
final byte[] iv = // retrieve from encryption step
final byte[] encrypted = // encrypted ciphertext

Key key = generateSecretKey(passphrase.toCharArray(), salt);
Cipher cipher = Cipher.getInstance("AES/CBC/PKCS5Padding");

cipher.init(Cipher.DECRYPT_MODE, key, new IvParameterSpec(iv));
byte[] plainText = cipher.doFinal(encrypted);
  • For more details, see Using Password-based Encryption on Android

Keystore and Keychain API

  • Since Donut/1.6, Android has had a system-wide keystore for storage of VPN and (later) WiFi authentication keys
    • See Working with secure certificates
    • Pre ICS/4.0, applications were not able to access it, so those needing to authenticate via client certificates ended up maintained their own key store (hard to manage across apps)
  • In ICS/4.0, Android adds support for android.security.KeyChain, which provides apps with
    • Access to private keys in the system key store and their corresponding certificate chains (subject to one-time user approval)
    • Ability to initiate installation of credentials from X.509 certificates and PKCS#12 key stores (e.g. for installation of organization CA certs)
  • Also, in ICS/4.0, Android uses the screen lock password to protect the system credential storage, which works well with device administration APIs and prevents the user from disabling the password as long as they use the secured credentials
  • See Unifying Key Store Access in ICS

Encryption Using Hardware-Backed Private Keys

  • On Android 4.3+, the hardware-backed AndroidKeyStore stores device-specific private keys.
    • See Using Android Keystore Provider
  • We can use these keys with the RSA cipher algorithm
  • RSA is an asymmetric-key algorithm, meaning it uses a separate "public" key to encrypt data and a "private" key to decrypt it.
  • RSA private keys are stored inside a secure hardware element.
    • Typically a Trusted Execution Environment (TEE) on the SoC.
    • KeyChain.isBoundKeyAlgorithm() will tell if other algorithms are also hardware-bound (and, therefore, non-exportable)
  • Advantage: This approach doesn’t require user input (password)
  • Disadvantage: Key generation is slow. Technically key is retrievable (though it would require dismantling the SoC)

Here is an example of using a non-exportable key to encrypt a chunk of data:

  1. To encrypt data with a hardware-bound key, we first need to generate one for the application: 

    //Create key pair with self-signed certificate information
Calendar cal = Calendar.getInstance();
Date now = cal.getTime();
cal.add(Calendar.YEAR, 1);
Date end = cal.getTime();

KeyPairGenerator kpg = KeyPairGenerator.getInstance("RSA", "AndroidKeyStore");
kpg.initialize(new KeyPairGeneratorSpec.Builder(context)
        .setAlias(KEY_ALIAS)
        .setStartDate(now)
        .setEndDate(end)
        .setSerialNumber(BigInteger.valueOf(1))
        .setSubject(new X500Principal("CN=" + KEY_ALIAS))
        .build());

//Calling this also binds the private key to hardware
// You could obtain it here with getPrivate(), if needed
KeyPair kp = kpg.generateKeyPair();

//Use public key to encrypt data
PublicKey publicKey = kp.getPublic();
    You only need to do this once, and persist the public key somewhere for re-use. Generating a hardware-bound key is expensive (many seconds).

  2. Now we can pass the public key to a Cipher: 

    byte[] plainText = // some plain text
Cipher cipher = Cipher.getInstance("RSA/ECB/PKCS1Padding");
cipher.init(Cipher.ENCRYPT_MODE, publicKey);
byte[] encrypted = cipher.doFinal(plainText);

  3. To decrypt, we retrieve the private key from the keystore: 

    KeyStore ks = KeyStore.getInstance("AndroidKeyStore");
ks.load(null);
KeyStore.Entry entry = ks.getEntry(KEY_ALIAS, null);
if (!(entry instanceof KeyStore.PrivateKeyEntry)) {
    throw new KeyException("Not an instance of a PrivateKeyEntry");
}

PrivateKey privateKey = ((KeyStore.PrivateKeyEntry) entry).getPrivateKey();

  4. The private key can then be passed to the decrypt Cipher: 

    byte[] encrypted = // encrypted ciphertext
Cipher cipher = Cipher.getInstance("RSA/ECB/PKCS1Padding");
cipher.init(Cipher.DECRYPT_MODE, privateKey);

byte[] plainText = cipher.doFinal(encrypted);

Symmetric Password-Based Encryption

Let’s focus on password encryption first. Inside PasswordEncryptor, there are three methods we need to fill in:

  • generateSecretKey() - Create a new key object from the password material.
  • encryptData() - Take plain text data and write it (encrypted) to a stream.
  • decryptData() - Take encrypted data from a stream and decrypt it.

The hooks for triggering this logic from the user interface are already in place:

  1. Implement generateSecretKey():

    1. Get an instance of a SecretKeyFactory using the "PBKDF2WithHmacSHA1" algorithm

    2. Create a new PBEKeySpec with the supplied password parameters

      Use an iterations count of at least 1000

    3. Generate a new SecretKey from the factory with generateSecret()

    4. Return a new SecretKeySpec containing the raw key bytes and algorithm: 

      return new SecretKeySpec(keyBytes, "AES");

  2. Implement encryptData() to write incoming data to a stream:

    1. Get a Cipher instance loaded with the "AES/CBC/PKCS5Padding" algorithm (already defined for your as ENCRYPTION_ALGORITHM)

    2. Use the provided SecureRandom to generate a random salt: 

      byte[] salt = new byte[SALT_LENGTH];
mSecureRandom.nextBytes(salt);

    3. Repeat the same process to create a new initialization vector (iv). The length of the array should be cipher.getBlockSize().

    4. Initialize the cipher with your key and iv: 

      Key key = generateSecretKey(passphrase.toCharArray(), salt);
cipher.init(Cipher.ENCRYPT_MODE, key, new IvParameterSpec(iv));

    5. Call doFinal() on the cipher to return an encrypted version of the incoming data bytes.

    6. We need the salt and iv again to decrypt the data. Write all three items (Base64-encoded) to supplied OutputStream.

    7. Write the supplied DELIMITER character in between each element so they can easily be split apart later.

      We chose & as the delimiter because it is a character not present in the Base64 encoding alphabet.

    8. flush() and close() the OutputStream

  3. Implement decryptData() to reverse the process:

    1. Use the supplied readFile() method to stream the encrypted data into memory as a string.

    2. Split the string on DELIMITER to get back the original three fields (salt, iv, encrypted data).

      Don’t forget to Base64 decode each field after splitting them!

    3. Init a new cipher for decryption using the same algorithm and key: 

      Key key = generateSecretKey(passphrase.toCharArray(), salt);
Cipher cipher = Cipher.getInstance(ENCRYPTION_ALGORITHM);

cipher.init(Cipher.DECRYPT_MODE, key, new IvParameterSpec(iv));

    4. Return the result of doFinal() to get back the decrypted note.

Using a Hardware-Bound RSA Key

Now let’s try this using a private key that is tucked away in a non-exportable block of the hardware. The RSAHardwareEncryptor has four methods you will need to implement:

  • retrievePublicKey() - Get the public (encryption) key from storage.
  • retrievePrivateKey() - Get the private (decryption) key from the keystore.
  • encryptData() - Take plain text data and write it (encrypted) to a stream.
  • decryptData() - Take encrypted data from a stream and decrypt it.
Hardware-backed key storage is only available on Android 4.3+

The hooks for triggering this logic from the user interface are already in place:

  1. Read through and take a moment to understand the generatePrivateKey() method.

    1. This method is responsible for creating the public/private key pair and persisting them both.

    2. The public key is stored to SharedPreferences.

    3. The private key is bound to the hardware keystore. It can be referenced later by its alias (KEY_ALIAS in this case).

  2. Implement retrievePublicKey() to create the encryption key:

    1. Retrieve the encoded key string from SharedPreferences.

    2. Decode the string with Base64.decode() into raw bytes.

    3. Use a KeyFactory to return a PublicKey objeect from the bytes: 

      return KeyFactory.getInstance(KEY_ALGORITHM)
                .generatePublic(new X509EncodedKeySpec(publicKey));

  3. Implement retrievePrivateKey() to get the decryption key from hardware:

    1. Get a KeyStore instance of the "AndroidKeyStore".

    2. Load the KeyStore with null as the InputStream parameter to initialize it.

    3. Get the KeyStore.Entry associated with our KEY_ALIAS from the KeyStore.

    4. Return the PrivateKey from the entry with getPrivateKey(): 

      return ((KeyStore.PrivateKeyEntry) entry).getPrivateKey();

  4. Implement encryptData() to write incoming data to a stream:

    1. Retrieve an instance of the public key.

    2. Initialize a new cipher in encrypt mode with the "RSA/ECB/PKCS1Padding" algorithm.

    3. Wrap the supplied OutputStream in a Base64OutputStream to encode all characters before file writing.

      Add the Base64.NO_WRAP flag to the stream to avoid reading conflicts.

    4. Wrap the encoding stream from the previous step in a CipherOutputStream to apply our cipher to all bytes written.

    5. Write the incoming data to our wrapped OutputStream, flush() and close().

  5. Implement decryptData() to reverse the process:

    1. Retrieve an instance of the private key.

    2. Initialize a new cipher in decrypt mode with the key and the same RSA algorithm.

    3. Wrap the provided InputStream in a Base64InputStream to decode the file bytes.

      Don’t forget the Base64.NO_WRAP flag again!

    4. Wrap the decoder stream in a CipherInputStream to apply decryption to the incoming data.

    5. Use the supplied readFile() helper to read data from the incoming stream

    6. Return the decoded data as a byte[].

      You can use getBytes() to return the bytes of a String.

  6. Run the application again. Type a new secret message to save.

  7. Select the Key-Based option, and click Save.

  8. If the save was successful, you can click Load and see your typed message in the lower box.
